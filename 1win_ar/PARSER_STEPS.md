# Пошаговое описание работы парсера 1win AR

## Общая схема работы

Парсер автоматизирует процесс получения платежных данных с сайта 1win.lat для Аргентины. Он использует Playwright для автоматизации браузера и извлекает данные о методе оплаты Claro Pay.

---

## Шаг 1: Инициализация браузера

**Файл:** `services/payment_parser_ar.py`, строки 79-165

1. **Создание persistent context** (если `use_persistent_context=True`):
   - Создает директорию для профиля браузера: `~/.cache/1win_ar/profile`
   - Запускает Chromium с persistent context (сохраняет cookies и сессию между запусками)
   - Настраивает параметры для обхода детекции автоматизации:
     - Отключает `AutomationControlled`
     - Устанавливает User-Agent: `Mozilla/5.0 (Macintosh; Intel Mac OS X 10_15_7)...`
     - Настраивает viewport: 1920x1080
     - Устанавливает timezone: `America/New_York`

2. **Удаление признаков автоматизации**:
   - Удаляет `navigator.webdriver`
   - Маскирует `navigator.permissions`
   - Маскирует `navigator.plugins`
   - Добавляет `window.chrome.runtime`

3. **Fallback на browser pool** (если persistent context не работает):
   - Использует общий пул браузеров

**Файл:** `services/payment_parser_ar.py`, строки 169-177

2. Ждет загрузки страницы (`domcontentloaded` или `load`)
3. Делает паузу 3 секунды для загрузки динамического контента

## Шаг 3: Проверка авторизации

**Файл:** `services/payment_parser_ar.py`, строки 179-200

   - Ищет элементы, указывающие на авторизацию (кнопка "Log in", форма входа)
   - Если пользователь уже авторизован и `skip_login=True`, пропускает шаг 4

## Шаг 4: Вход в систему (если не авторизован)

**Файл:** `services/payment_parser_ar.py`, строки 202-776

### 4.1. Поиск формы входа

1. **Метод 1:** Ищет модальное окно входа:
   - Селекторы: `div[role="dialog"]`, `[class*="modal" i]`, `[class*="login" i]`
   - Ищет кнопку "Log in" или "Login"

2. **Метод 2:** Ищет форму входа на странице:
   - Селекторы: `form`, `input[type="email"]`, `input[type="password"]`

### 4.2. Заполнение формы

1. Находит поле email:
   - Селекторы: `input[type="email"]`, `input[name*="email" i]`, `input[placeholder*="email" i]`
   - Вводит email

2. Находит поле password:
   - Селекторы: `input[type="password"]`
   - Вводит пароль

### 4.3. Отправка формы

1. **Метод 1:** Ищет кнопку по тексту:
   - Тексты: "Log in", "Login", "Войти", "Ingresar", "Entrar", "Sign in"
   - Кликает на кнопку

2. **Метод 2:** Ищет кнопку `button[type="submit"]`

3. **Метод 3:** Нажимает Enter на поле пароля

4. Ждет 3 секунды после отправки формы

---

## Шаг 5: Обработка промо-редиректов

**Файл:** `services/payment_parser_ar.py`, строки 782-797

1. Проверяет URL на наличие промо-параметров:
   - `trid=` в URL
   - `banner` в URL

2. Если обнаружен промо-редирект:
   - Переходит на `about:blank`
   - Переходит обратно на `base_url`
   - Если промо все еще есть, кликает на логотип (ссылку на главную)

---

## Шаг 6: Переход на страницу депозита

**Файл:** `services/payment_parser_ar.py`, строки 778-867

1. **Приоритет 1:** Прямой переход на `/deposit`:
   - Переходит на `https://1win.lat/deposit`
   - Проверяет, не произошел ли редирект на промо
   - Если успешно, переходит к шагу 7

2. **Приоритет 2:** Поиск кнопки "Deposit" по тексту:
   - Тексты: "Deposit", "Depositar", "Depósito", "Пополнить"
   - Кликает на найденную кнопку

3. **Приоритет 3:** Поиск кнопки через CSS селекторы:
   - `[href*="deposit" i]`
   - `[data-testid="deposit"]`
   - `a[href*="/deposit"]`
   - `button[data-action="deposit"]`

4. Ждет 3 секунды после клика/перехода

## Шаг 7: Обработка промо-редиректа после депозита

**Файл:** `services/payment_parser_ar.py`, строки 869-920

1. Проверяет, не произошел ли редирект на промо после перехода на депозит

2. Если промо обнаружено:
   - **Метод 1:** Пытается открыть модальное окно через JavaScript:
     - Ищет `div.paywidget-app` в DOM
     - Пытается изменить `display: none` на `display: block`
     - Или кликает на кнопку депозита через JS
   
   - **Метод 2:** Возвращается на главную и снова кликает на "Deposit"

3. Ждет 5 секунд для загрузки модального окна

## Шаг 8: Поиск модального окна оплаты

**Файл:** `services/payment_parser_ar.py`, строки 922-902

1. Ищет модальное окно `div.paywidget-app`

2. Если не найдено, пробует альтернативные селекторы:
   - `[class*='paywidget']`
   - `[class*='payment']`
   - `[class*='deposit']`
   - `div[role='dialog']`

3. Если модальное окно не найдено:
   - Делает скриншот для диагностики: `modal_not_found.png`
   - Логирует ошибку

## Шаг 9: Поиск и клик на Claro Pay

**Файл:** `services/payment_parser_ar.py`, строки 904-1095

### 9.1. Поиск кнопок методов оплаты

1. **Метод 1:** Ищет кнопки через `data-testid`:
   - Селектор: `button[data-testid^="payment-method-"]`
   - Ждет появления хотя бы одной кнопки (до 15 секунд)

2. Если кнопки не найдены, пробует альтернативные селекторы:
   - `button[data-testid*="payment"]`
   - `button[class*="payment"]`
   - `div[class*="payment-method"]`

3. Получает все найденные кнопки

### 9.2. Поиск Claro Pay

1. Перебирает все кнопки методов оплаты

2. Пропускает криптовалютные методы (если `crypto` в `data-testid`)

3. Ищет название метода:
   - Селектор: `span._label_1obhu_10` (внутри кнопки)
   - Проверяет, содержит ли текст "Claro Pay" или "claro pay"

4. Если найден:
   - Прокручивает кнопку в видимую область
   - Кликает на кнопку
   - Ждет 3 секунды
   - Логирует успех

### 9.3. Fallback методы поиска

Если не найден через `data-testid`:

1. **Метод 2:** Поиск по тексту "Claro Pay" на странице
2. **Метод 3:** Поиск через CSS селекторы с текстом "Claro Pay"

### 9.4. Скриншот после клика

1. Если Claro Pay найден и кликнут:
   - Ждет 2 секунды
   - Делает скриншот страницы
   - Сохраняет в: `~/.cache/1win_ar/screenshots/provider_claro_YYYYMMDD_HHMMSS.png`
   - Сохраняет base64 в результат

## Шаг 10: Выбор типа Fiat и нажатие Deposit

**Файл:** `services/payment_parser_ar.py`, строки 1119-1234

### 10.1. Проверка раннего успеха

1. Проверяет, появилась ли форма сразу:
   - Ищет текст: `"2. Enter payment details"`
   - Если найдено, переходит к шагу 11

### 10.2. Выбор типа Fiat

1. **Метод 1:** Поиск по тексту "Fiat":
   - Ищет элемент с текстом "Fiat"
   - Кликает на него

2. **Метод 2:** Поиск через CSS селекторы:
   - `[data-type="fiat"]`
   - `.fiat-tab`
   - `#fiat-tab`
   - `[class*="fiat" i]`
   - `button:has-text("Fiat")`

3. Ждет 2 секунды после клика

### 10.3. Нажатие кнопки Deposit

1. Ищет кнопку `button[data-testid="deposit-button"]`

2. Если не найдена, ищет по тексту "deposit" или "depositar"

3. Кликает на кнопку

### 10.4. Ожидание формы с данными

1. Ждет до 20 секунд появления формы

2. Проверяет индикаторы успешной загрузки:
   - Текст "Error" → ошибка
   - Текст "Payment failed" → ошибка
   - `img[alt="qrCode"]` → QR код найден
   - Текст "We're waiting for your transfer:" → форма загружена
   - Наличие текста "CVU", "Recipient" или "Bank" в теле страницы

3. Если форма появилась, переходит к шагу 11

## Шаг 11: Извлечение данных платежной формы

**Файл:** `services/payment_parser_ar.py`, строки 1242-1440

### 11.1. Подготовка

1. Ждет 5 секунд для полной загрузки формы
2. Сохраняет текущий URL в результат
3. Делает скриншот страницы (для отладки)

### 11.2. Извлечение CVU

**Регулярные выражения:**
- `CVU[:\s]*([0-9]{20,25})`
- `CVU[:\s]*([\d\s\-]{20,30})`
- `(?:CVU|Alias)[:\s]*([0-9]{20,25})`
- `([0-9]{20,25})` (fallback)

**Методы поиска:**

1. **Метод 1:** Поиск в HTML контенте через regex
2. **Метод 2:** Поиск в видимом тексте страницы через regex
3. **Метод 3:** Поиск через DOM селекторы:
   - `*:has-text("CVU")`
   - `*:has-text("CVU") + *` (следующий элемент)
   - `[data-field*="cvu" i]`
   - `[name*="cvu" i]`
   - `[id*="cvu" i]`
   - `[class*="cvu" i]`
   - И другие...

4. Валидация: CVU должен быть длиной 20-25 символов

### 11.3. Извлечение Recipient (Получатель)

**Методы поиска:**

1. **Метод 1:** Поиск через DOM селекторы:
   - `*:has-text("Recipient")`
   - `*:has-text("Recipient") + *`
   - `[data-field*="recipient" i]`
   - `[name*="recipient" i]`
   - И другие...

2. **Метод 2:** Поиск через regex в тексте:
   - `Recipient[:\s]*([A-Za-z\s]+)`

3. Извлекает текст из найденного элемента

### 11.4. Извлечение Bank (Банк)

1. Ищет текст "Bank" или "Banco"
2. Извлекает название банка из соседних элементов
3. Если не найден, использует "Claro Pay" как значение по умолчанию

### 11.5. Извлечение Amount (Сумма)

1. Ищет сумму через regex:
   - `Amount[:\s]*([\d,.\s]+)`
   - `([\d,.\s]+)\s*(?:ARS|USD|EUR)`

2. Ищет через DOM селекторы с текстом "Amount"

3. Очищает от пробелов и форматирует

### 11.6. Определение Method и Payment Type

1. **Method:** "Claro Pay" (если кнопка была найдена)
2. **Payment Type:** "Fiat" (если был выбран тип Fiat)

### 11.7. Извлечение Domain

1. Извлекает домен из URL: `1win.lat`

## Особенности и обходы проблем

1. **Промо-редиректы:** Парсер автоматически обнаруживает и обрабатывает редиректы на промо-страницы
2. **Persistent context:** Сохраняет cookies между запусками, позволяя пропускать логин
3. **Множественные методы поиска:** Для каждого элемента используется несколько методов поиска (fallback)
4. **Ожидания:** Используются таймауты и ожидания для динамического контента
5. **Скриншоты:** Сохраняются на каждом этапе для диагностики
6. **Логирование:** Подробное логирование всех действий через structlog

---

## Текущие проблемы

1. **CVU, Recipient, Amount не извлекаются:**
   - Причина: Парсер не доходит до формы с данными (редирект на промо-страницу)
   - URL остается: `https://1win.lat/promotions?trid=banner1wBonusUpTo600INbr`
   - Модальное окно с методами оплаты не открывается

2. **Claro Pay не находится:**
   - Модальное окно `div.paywidget-app` не появляется после клика на Deposit
   - Возможные причины:
     - Промо-редирект блокирует открытие модального окна
     - Модальное окно открывается с задержкой
     - Селекторы модального окна изменились

---

## Рекомендации по улучшению

2. Добавить больше альтернативных селекторов для модального окна

5. Использовать headless=False для отладки (временно)
