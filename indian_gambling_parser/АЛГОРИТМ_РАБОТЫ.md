# Пошаговый алгоритм работы парсера провайдеров 1xbet

## Общая схема

Парсер автоматически находит провайдеров платежей (UPI Fast и Paytm) на сайте 1xbet, взаимодействует с ними, заполняет формы и сохраняет данные о доменах провайдеров и скриншоты страниц.

---

## Детальный пошаговый алгоритм

### Фаза 1: Инициализация и логин

#### Шаг 1.1: Запуск браузера
- Парсер запускает браузер Chromium (Playwright)
- Загружает сохранённый `storage_state` (если есть) для быстрого логина
- Создаёт новый контекст браузера с настройками (user-agent, viewport, и т.д.)

#### Шаг 1.2: Загрузка главной страницы
- Переходит на URL мерчанта: `https://indian.1xbet.com`
- Ожидает загрузки страницы (timeout: 60 секунд)
- Обрабатывает ошибки загрузки с fallback-вариантами

#### Шаг 1.3: Автоматический логин
- Пытается найти поля username/password по селекторам из конфига
- Заполняет поля и нажимает кнопку входа
- Если логин успешен → сохраняет `storage_state` для следующих запусков

#### Шаг 1.4: Обработка капчи (Human-in-the-loop)
- Если обнаружена капча или логин не удался:
  - Открывает браузер (не headless)
  - Печатает инструкции в консоль
  - **Останавливается и ждёт ручного ввода**
  - Пользователь должен:
    1. Решить капчу в браузере (если есть)
    2. Ввести креденшиалы
    3. Войти в систему
    4. Нажать Enter в терминале
  - После успешного логина сохраняет `storage_state`

---

### Фаза 2: Переход на страницу кассира (Cashier)

#### Шаг 2.1: Навигация на страницу депозитов
- Парсер переходит на URL кассира: `https://indian.1xbet.com/en/office/recharge`
- Если URL не указан в конфиге, ищет кнопку "Касса" или "Deposit" на странице

#### Шаг 2.2: Нажатие кнопки "Make Deposit"
- Ищет кнопку "Make Deposit" / "Пополнить счёт" по селекторам из конфига
- Кликает по кнопке
- Ожидает загрузки формы депозита

---

### Фаза 3: Поиск секции E-WALLETS

#### Шаг 3.1: Определение контекста (страница или iframe)
- Парсер ищет текст "E-WALLETS" на главной странице
- Если не найден, проверяет все iframe на странице
- Определяет, где находится секция E-WALLETS (page или frame)

#### Шаг 3.2: Прокрутка к секции E-WALLETS
- Находит заголовок "E-WALLETS" в найденном контексте
- Прокручивает к секции (scroll_into_view_if_needed)
- Ожидает загрузки контента (timeout: 3 секунды)

---

### Фаза 4: Поиск провайдеров UPI Fast и Paytm

#### Шаг 4.1: Поиск по изображениям
- Парсер ищет изображения с `alt` или `src`, содержащими "upi fast" или "paytm"
- Собирает все найденные изображения

#### Шаг 4.2: Поиск по тексту (fallback)
- Если изображения не найдены, ищет элементы с текстом:
  - "UPI Fast" (регистронезависимо, с учётом пробелов)
  - "Paytm" / "PayTM" (регистронезависимо)

#### Шаг 4.3: Определение кликабельной карточки
- Для каждого найденного элемента находит родительский кликабельный элемент
- Ищет предка с тегом `a`, `button`, или с атрибутами `role="button"`, `onclick`
- Это гарантирует клик именно по карточке, а не по тексту

#### Шаг 4.4: Фильтрация только UPI Fast и Paytm
- Оставляет только элементы, соответствующие "UPI Fast" или "Paytm"
- Сохраняет список элементов для обработки

---

### Фаза 5: Обработка каждого провайдера (цикл)

Для каждого провайдера (UPI Fast, затем Paytm):

#### Шаг 5.1: Диагностика элемента
- Проверяет видимость элемента (is_visible)
- Получает bounding_box элемента
- Выводит диагностическую информацию

#### Шаг 5.2: Клик по карточке провайдера
- Прокручивает к элементу (scroll_into_view_if_needed)
- Пытается кликнуть через `elem.click(force=True)`
- Если не получается, использует `page.mouse.click` с координатами
- Ожидает изменения UI (timeout: 8 секунд)

#### Шаг 5.3: Ожидание платёжного UI
- Парсер ждёт появления платёжной формы:
  - Поля ввода суммы (input[name*='amount'])
  - Кнопки "Continue", "Pay", "Deposit", "Submit"
  - Текста "Enter amount", "UPI", "VPA"
  - Поля для UPI ID, phone, и т.д.
- Если UI не появился, продолжает работу

#### Шаг 5.4: Заполнение формы в попапе
- Парсер находит кнопку "ПОДТВЕРДИТЬ" / "CONFIRM" (по regex)
- Определяет, какое поле нужно заполнить:
  1. **UPI ID** (приоритет 1):
     - Ищет поле по label с текстом "upi id" (regex, case-insensitive)
     - Или по input[name*='upi' i], input[placeholder*='upi' i]
     - Заполняет значением из конфига: `kapil.ss17@icici`
  
  2. **Phone Number** (приоритет 2, если UPI не найден):
     - Ищет поле по label с текстом "number|phone|mobile" (regex)
     - Или по input[type='tel'], input[name*='phone' i]
     - Заполняет значением из конфига: `+918925242020`
  
  3. **Aadhaar** (приоритет 3, если остальные не найдены):
     - Ищет поле по label с текстом "aadhaar" (regex)
     - Или по input[name*='aadhaar' i]
     - Заполняет значением из конфига: `571517312306`

- **Важно**: Значения берутся из `config["provider_form_data"]`, не логируются, не хардкодятся

#### Шаг 5.5: Нажатие кнопки "ПОДТВЕРДИТЬ"
- Находит кнопку подтверждения (get_by_role("button", name=re.compile(r"подтвердить|confirm", re.I)))
- Кликает по кнопке (force=True, timeout: 8 секунд)
- Ожидает редиректа на домен провайдера

---

### Фаза 6: Определение домена провайдера (после confirm)

**Критически важно**: Провайдером считается только домен, который появился **после** нажатия "ПОДТВЕРДИТЬ".

#### Шаг 6.1: Ожидание внешнего домена
- Парсер ждёт до 12 секунд появления внешнего домена
- Слушает network requests (фильтрует tracking-домены)
- Проверяет URL всех iframe
- Использует функцию `wait_external_after_confirm()`

#### Шаг 6.2: Фильтрация tracking-доменов
- Исключает домены мерчанта (1xbet.com, indian.1xbet.com)
- Исключает tracking-домены:
  - google.com, google.co.in, google-analytics.com
  - facebook.com, fbcdn.net
  - yandex.ru, yandex.net
  - doubleclick.net, googlesyndication.com
  - И другие известные tracking-сервисы

#### Шаг 6.3: Поиск полного URL провайдера
Парсер ищет полный URL в следующем порядке приоритета:

1. **Iframe** (приоритет 1):
   - Проверяет все iframe на странице
   - Ищет iframe с доменом, совпадающим с найденным провайдером
   - Если найден → использует URL этого iframe

2. **Navigation** (приоритет 2):
   - Проверяет текущий URL страницы (page.url)
   - Если домен совпадает с провайдером → использует этот URL

3. **Network request** (приоритет 3):
   - Ищет в собранных network requests
   - Находит первый запрос с доменом провайдера
   - Использует URL этого запроса

#### Шаг 6.4: Проверка и валидация
- Если провайдер не найден → **пропускает этот провайдер**, переходит к следующему
- Если найден провайдер, но это не внешний домен → **пропускает**
- Если найден валидный провайдер → продолжает

### Фаза 7: Создание скриншота

#### Шаг 7.1: Определение объекта для скриншота
- Приоритет 1: Если найден iframe провайдера → делает скриншот iframe
- Приоритет 2: Если открылся popup → делает скриншот popup
- Приоритет 3: Если редирект на страницу → делает скриншот страницы

#### Шаг 7.2: Сохранение скриншота
- Имя файла: `{merchant_domain}_{provider_domain}_{timestamp}.png`
- Путь: `screenshots/{filename}`
- Скриншот делается **после confirm**, то есть это скриншот страницы провайдера, а не попапа

### Фаза 8: Сохранение данных

#### Шаг 8.1: Подготовка данных
Парсер собирает следующую информацию:
- `merchant_id`: "1xbet"
- `merchant_domain`: "indian.1xbet.com"
- `provider_domain`: домен провайдера (появившийся после confirm)
- `provider_name`: название провайдера ("UPI Fast" или "Paytm")
- `account_type`: "player" (из конфига)
- `payment_method`: "e-wallet" или "upi" (определяется автоматически)
- `final_url`: полный URL провайдера
- `entry_url`: URL входа на страницу провайдера
- `screenshot_path`: путь к скриншоту
- `detected_in`: способ обнаружения ("iframe", "navigation", "network")
- `is_iframe`: булево значение (True если провайдер в iframe)
- `timestamp`: текущее время

#### Шаг 8.2: Проверка дубликатов
- Проверяет, существует ли уже запись с таким `merchant_domain` и `provider_domain`
- Если существует → пропускает, переходит к следующему провайдеру

#### Шаг 8.3: Сохранение в SQLite
- Сохраняет данные в локальную БД: `providers_data.db`
- Таблица: `providers`

#### Шаг 8.4: Экспорт в Google Sheets
- Экспортирует данные в Google Sheets:
  - Google Sheet ID: `1kw74QoJGHKlaLOD-FGEPRU0DXKOAh4S-8_1Bz6burFE`
  - Использует credentials из `google_credentials.json`
  - Очищает старые данные (кроме заголовков)
  - Загружает все данные из БД

### Фаза 9: Возврат на страницу кассира (для следующего провайдера)

#### Шаг 9.1: Возврат на страницу депозитов
- Парсер переходит обратно на `cashier_url`: `https://indian.1xbet.com/en/office/recharge`
- Ожидает загрузки страницы

#### Шаг 9.2: Повторное нажатие "Make Deposit"
- Если нужно, нажимает кнопку "Make Deposit" снова
- Ожидает загрузки формы депозита

#### Шаг 9.3: Переход к следующему провайдеру
- Возвращается к **Шагу 5.1** (обработка следующего провайдера)
- Повторяет процесс для следующего элемента (UPI Fast или Paytm)

#### Шаг 10.1: Закрытие браузера

#### Шаг 10.2: Финальный экспорт

## Ключевые особенности алгоритма

### 1. Провайдер определяется только после confirm
- Провайдером считается **только** домен, который появился после нажатия "ПОДТВЕРДИТЬ"
- Все домены, появившиеся до confirm, игнорируются
- Если внешний домен не появился после confirm → провайдер пропускается

### 2. Безопасная обработка чувствительных данных
- UPI ID, Phone Number, Aadhaar берутся из конфига, не хардкодятся
- Значения не логируются в консоль
- Скриншоты не маскируются (можно добавить опционально)

### 3. Надёжный поиск элементов
- Поиск по изображениям (приоритет 1)
- Поиск по тексту с regex (fallback)
- Поиск кликабельного родителя (карточки)
- Диагностика перед кликом

### 4. Обработка ошибок
- Все критические операции обёрнуты в try-except
- Fallback-варианты для загрузки страниц
- Пропуск провайдеров при ошибках (продолжение работы)

### 5. Фильтрация tracking-доменов
- Исключение известных tracking-сервисов
- Исключение доменов мерчанта
- Приоритет iframe > navigation > network requests

## Пример выполнения для 1xbet

1. **Запуск**: `python3 main_parser_playwright.py --merchant 1xbet --url https://indian.1xbet.com`

2. **Логин**: Автоматический (или ручной через human-in-the-loop)

3. **Переход на кассир**: `https://indian.1xbet.com/en/office/recharge`

4. **Поиск провайдеров**: 
   - Находит "UPI Fast" и "Paytm" в секции E-WALLETS

5. **Обработка UPI Fast**:
   - Клик по карточке → открывается попап
   - Заполнение UPI ID: `kapil.ss17@icici`
   - Нажатие "ПОДТВЕРДИТЬ"
   - Редирект на домен провайдера (например: `payments.example.com`)
   - Скриншот страницы провайдера
   - Сохранение данных

6. **Возврат на кассир**: Переход обратно на страницу депозитов

7. **Обработка Paytm**:
   - Клик по карточке → открывается попап
   - Заполнение Phone Number: `+918925242020`
   - Нажатие "ПОДТВЕРДИТЬ"
   - Редирект на домен провайдера (например: `paytm.com`)
   - Скриншот страницы провайдера
   - Сохранение данных

8. **Завершение**: Экспорт в Google Sheets, закрытие браузера

## Результаты работы

После выполнения парсер сохраняет:

1. **Локальная БД** (`providers_data.db`):
   - Все найденные провайдеры с метаданными
   - Скриншоты (пути к файлам)

2. **Google Sheets** (`1kw74QoJGHKlaLOD-FGEPRU0DXKOAh4S-8_1Bz6burFE`):
   - Та же информация в табличном виде
   - Обновляется после каждого провайдера

3. **Скриншоты** (`screenshots/`):
   - PNG-файлы страниц провайдеров
   - Имена: `{merchant}_{provider}_{timestamp}.png`
